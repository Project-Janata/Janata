Product Development Lanes

Keep product scope small enough that beta feedback changes behavior, not just backlog priority.

1. Trust and access

  • Verification tiers, email verification, invite redemption, and clear unverified states.
  • Make invite generation usable by real CHYK coordinators; test with cross-device links.
  • Turn GitHub #201, #202, #204 into explicit QA scripts rather than leaving them as broad feature issues.

2. Discovery and participation

  • Explore must make local centers and events obvious without demo narration.
  • Feed should show public posts and board posts reliably, but public posting must not risk data loss in migration.
  • Presentation demo should show: find center, find event, join, post, reply, invite another CHYK.

3. Operations and moderation

  • Moderation/admin tools can be minimal, but there must be a credible response path for bad posts.
  • Support inbox/alias is required before broader TestFlight distribution.
  • Metrics should capture signup source, invite conversion, center joins, event taps, and retained beta users.

iOS Submission Plan

Notion says the Apple account path is resolved through CCMT; treat that as a coordination task, not an excuse to delay build readiness.

Submission checklist

  • Confirm bundle identifier and ownership path: current app id is com.theabhiramr.project-janata; decide transfer vs fresh CCMT app.
  • Run EAS production build under the account that will own TestFlight distribution.
  • Prepare App Store metadata: description, screenshots, privacy nutrition labels, support URL, age rating, contact.
  • Configure universal links / associated domains if invite links are part of the beta path.
  • Submit TestFlight build early enough for beta review and external tester approval.

Beta feedback loop

  • Start with San Jose pilot in early June: one center, one coordinator, one invite cohort.
  • Collect feedback in three buckets: unable to sign in, unable to find/join, unclear why to return.
  • Weekly beta build cadence: Monday triage, Tuesday fixes, Wednesday internal QA, Thursday TestFlight, weekend observation.
  • By July 13, only fix launch-blocking issues; move feature requests to post-MSC.
  • Prepare a fallback web demo in case App Review timing slips.
